# Verdict — диагностика `color_classifier_v4` на cam-13

**Дата:** 2026-04-25
**Видео:** `recordings/yaris_20260303_164835/kamera_13_164841_END165759.mp4` (9.3 мин, 1280×720, 25 fps)
**Модель:** `models/color_classifier_v4.pt` / `.onnx` / `.engine` (прод-веса, не трогали)

---

## 1. Проблема

`color_classifier_v4` **систематически уверенно ошибается** на людях, которых не должно быть в кадре (судья / оператор / посторонний в тёмной одежде на старте), и на крупных не-человеческих объектах (КАМАЗ). Ошибки не случайные — это устойчивый паттерн с высокой confidence.

В прод-сценарии это означает: бэкенд получает «green jockey @ 0.82» там где жокея нет, и кормит этим TimeTracker → ranking. На full-loop записи (см. memory `project_pipeline_false_positives.md`) это даёт 56% мусора в детекциях, 98% которых не переживают фильтр `h≥100 ∧ conf≥0.7`. Сейчас мы доказали, на каком именно механизме модели это построено.

---

## 2. Доказательная база

### Confusion matrix (420 размеченных crops с cam-13)

| true \ pred | blue | green | purple | red | yellow | total | accuracy |
|---|---:|---:|---:|---:|---:|---:|---:|
| **green** (real)      | 0 | **55** | 0 | 0 | 0 | 55 | **100%** ✅ |
| **yellow** (real)     | 0 | 0 | 0 | 0 | **56** | 56 | **100%** ✅ |
| **red** (real)        | 0 | 0 | 0 | **59** | 0 | 59 | **100%** ✅ |
| **not_jockey**        | 0 | 233 | 11 | 0 | 6 | 250 | **0%** ❌ |

На реальных жокеях **100% точность** (170/170). Не-жокеи **поголовно** мапятся в один из 5 классов — модель не имеет способа сказать «здесь нет жокея».

### Распределение `not_jockey` predictions

- **93.2%** → `green` (доминирующий backgroud cam-13 = газон)
- **4.4%** → `purple` (тени, тёмное оборудование)
- **2.4%** → `yellow` (КАМАЗ-эпизод, frames 002580..002730 — 6 секунд непрерывного попадания)
- **0%** → `blue` или `red` (нет таких доминантных цветов в кадре)

### Confidence distribution на not_jockey
- median = **0.82**, mean = **0.79**, max = **0.96**
- **21.6% (54 crops)** имеют conf ≥ 0.9 — соответствует исходному наблюдению пользователя «модель уверенно (>0.9) выдаёт green на не-жокее»
- **81.6%** имеют conf ≥ 0.7
- **0%** ниже 0.3 — модель **никогда не сомневается**

### Grad-CAM наблюдения (визуально, из `gallery_gradcam.html`)

1. На `not_jockey → green`: heatmap красный на **фоне (трава/земля)**, синий на торсе человека. Модель смотрит на окружение, не на одежду.
2. При смешанном фоне (жёлтый кузов КАМАЗа + тёмное) → **`yellow`** с conf 0.83. Модель усредняет цвет всего crop'а, не локализует одежду.
3. На real jockeys (control) — heatmap локализован на торсе с silk'ом. То есть модель **умеет** смотреть на одежду — но **только когда фон уже схож с одеждой по цвету**, и она не различает что доминирует.

## 3. Корневая причина (mechanism, не «модель плохая»)

Сломано **не одно, а два связанных места**:

### 3.1 Архитектурное усреднение (`SimpleColorCNN.features[9]`)

```python
self.features = nn.Sequential(
    Conv2d(3,32,3) → ReLU → MaxPool(2),
    Conv2d(32,64,3) → ReLU → MaxPool(2),
    Conv2d(64,128,3) → ReLU → MaxPool(2),
    AdaptiveAvgPool2d(4),    # ← сжимает 16×16 spatial map в 4×4 средними
)
```

`AdaptiveAvgPool2d(4)` — **глобальное пространственное усреднение** активаций. Сеть лишена возможности «выделить торс и проигнорировать фон» — она вынуждена учиться **среднему цвету всего crop'а**. Любые локальные spatial-признаки (где находится silk внутри crop'а) теряются перед `Linear(2048, 256)`.

Это объясняет heatmap-наблюдения: Grad-CAM на `features[6]` (последний Conv) показывает что **активации сильны там, где есть доминирующий цвет**, а downstream pooling забирает их все одинаково.

### 3.2 Отсутствие класса `not_jockey` в обучающем датасете

`models/color_classifier_v4.pt` обучен на 5-class softmax = `{blue, green, purple, red, yellow}`. **Нет класса «не жокей» / «нет уверенности»**. Softmax — это choose-one — он МАТЕМАТИЧЕСКИ обязан положить вероятность ≥0.2 в какой-то из 5 классов.

В тренировочном наборе (`data/color_dataset/`, см. ранний анализ): **725 crops, blue=15, purple=33** — крайний дисбаланс, и **0 примеров «фон без жокея»**. Сеть учится «вход всегда содержит жокея, угадай какого цвета» — и в проде с уверенностью применяет это правило к чему попало.

### 3.3 Связка двух дефектов = «доминирующий цвет crop'а → класс»

Architectural averaging + missing reject-class совместно дают чистое отображение:
> **«Какой цвет занимает больше всего пикселей в crop'е → этот класс с высокой confidence»**

Поэтому: трава → green, КАМАЗ → yellow, тёмное → purple. Это **не баг отдельной модели — это следствие выбранной архитектуры и обучающей задачи**. Не «модель плохо обучена» — она **обучена правильно делать неправильную вещь**.

## 4. Три варианта починки

### Вариант A — Tighter crops (input-side fix, без ретрейна)

**Идея:** уменьшить долю фона в crop'е, оставив только торс с silk. Делается на уровне SGIE input window: вместо «person bbox целиком» вырезать только `[y1 + 0.15·h, y1 + 0.50·h]` (верхняя торсовая половина, которая на лошади всегда содержит silk).

| Параметр | Значение |
|---|---|
| **Время реализации** | 1-2 часа: правка `deepstream/configs/sgie_color.txt` (`process-mode=2` уже crop по bbox) → нужен custom upstream cropper или ROI-сжатие в C++ probe |
| **Эффект** | Снижает фон в crop'е на ~50%. False positive rate упадёт, но не до 0: КАМАЗ всё ещё займёт верхнюю половину, газон позади жокея на дальних камерах тоже. |
| **Риск** | **Low.** Если торс-window слишком узкий — теряем информацию о цвете на real jockeys (особенно при низких/боковых ракурсах). Нужна A/B-проверка через `tools/run_record.py` на эталонной записи. |
| **Ничего не ретрейнится** | ✅ Прод-веса не трогаем |

### Вариант B — Retrain с классом `not_jockey` (data fix, та же архитектура)

**Идея:** добавить 6-й класс `not_jockey` (~300-500 примеров: газон, КАМАЗ, судья, оператор, тени, столбы) к существующим 5. Архитектура не меняется, добавляется один выходной нейрон.

| Параметр | Значение |
|---|---|
| **Время реализации** | 1-2 дня: (1) auto-extract not_jockey crops из run_record full_loop детекций где `pred=green ∧ conf≥0.7 ∧ filter h<60` или ROI-вне → ~1000 кандидатов, (2) ручная чистка ~30 мин, (3) re-balance датасета (blue=15→100, purple=33→100), (4) ретрейн → новый `.pt` → ONNX → TRT engine. |
| **Эффект** | Не-жокеи получают `not_jockey` softmax doминантным. В прод-pipeline добавляется одна строка: `if pred == "not_jockey": skip`. **Закроет ~85-90%** false positives. Оставшиеся — в graybar (mixed scene) — упадут до conf < 0.5 если confidence-threshold подкрутить. |
| **Риск** | **Medium.** Если not_jockey датасет смещён (только cam-13 трибуны) → модель может «зеленить» новые виды фона. Нужна валидация на full_loop с разными камерами. Архитектурная проблема averaging остаётся — но симптом локализован. |
| **Ретрейнится** | ✅ Тот же `SimpleColorCNN`, новые 6 классов |

### Вариант C — Retrain с attention pooling + not_jockey (архитектура + данные)

**Идея:** заменить `AdaptiveAvgPool2d(4)` на attention-based pooling (сеть учится **сама** взвешивать spatial regions: «эти пиксели важны, эти — фон»). Простейшая форма — Squeeze-and-Excitation или 1×1 conv → softmax → weighted sum. Плюс не-жокей класс из варианта B.

| Параметр | Значение |
|---|---|
| **Время реализации** | 3-5 дней: (1) изменить `SimpleColorCNN` (~30 строк), (2) собрать датасет как в варианте B, (3) ретрейн с attention head, (4) валидация что attention действительно фокусируется на торсе (повторить наш Grad-CAM пайплайн), (5) ONNX → TRT (новый ONNX opset, проверка что custom parser в `color_classifier_parser.cpp` совместим). |
| **Эффект** | Решает корневую проблему — модель учится **локализовать** silk вместо усреднения. Накладывается с вариантом B → **>95%** правильности на mixed-scene crops. Бонус: повышает точность на real jockeys в сложных ракурсах (со спины, частично закрытые). |
| **Риск** | **Higher.** (a) Custom parser в C++ (`NvDsInferClassiferParseCustomColor` в `libnvdsinfer_racevision.so`) может потребовать правки если output shape поменяется — а это HARD-зона из CLAUDE.md. (b) Attention pooling переобучается легче на маленьком датасете → нужно больше данных + аугментация. (c) Прод-conversion .pt→ONNX→TRT может выявить opset/op-compatibility issues. |
| **Ретрейнится + меняется architecture** | ⚠️ Затрагивает `models/`, `deepstream/configs/sgie_color.txt`, **возможно** C++ parser |

## 5. Рекомендация

**Делать в порядке D → A → B, вариант C — только если B недостаточно.**

### Почему именно так

0. **Вариант D (HSV agreement) — быстрейший first shot, делать ПЕРЕД A/B/C.** Дешёвый (1-2 вечера), полностью обратимый (один if в C++ parser, отключается флагом), измеримый на существующем размеченном наборе (420 crops с ground truth дают сразу precision/recall). Не требует ретрейна, не трогает прод-веса. Идеальный «попробуй сначала это» — даже если потом всё равно делаем A/B, базовая HSV-страховка останется как страховочный слой.

1. **Вариант A (tighter crops) — вторым.** За 1-2 часа получаем измеримое улучшение без риска для прод-весов. Если эффект существенный (false positive rate падает до приемлемого) — можем остановиться здесь и забыть про ретрейн на ближайшее время. Если эффект слабый — у нас уже есть baseline-метрики для следующего шага.

2. **Вариант B (retrain c not_jockey) — основное долгосрочное решение.** Это **минимальное** изменение которое лечит корневую причину «модель не умеет сказать нет». Архитектура не меняется → весь прод-pipeline (ONNX, TRT engine, C++ parser, SGIE config) совместим без правок. Класс `not_jockey` мапится в SHM как color_id=5 (или сохраняем 5-class но с дополнительной reject-confidence) — surgical change.

3. **Вариант C (architecture) — только если B недостаточно.** Architecture change + ретрейн + возможные правки HARD-зоны (`deepstream/src/color_classifier_parser.cpp`) — это сильно дороже и рискованнее. Имеет смысл только если после варианта B остаются специфические сценарии (mixed-color scenes на dense-камерах cam-01..04 где трибуны просвечивают сквозь жокеев).

### Конкретный первый шаг

**Реализовать вариант A** на `tools/run_record.py` сначала (там можно протестировать сужение crop input в Python без правки прод C++). Сравнить confusion matrix на текущих 250 not_jockey crops c таким же сужением. Если `not_jockey → green` падает с 93.2% до <40% — переносить в SGIE C++ parser. Если не падает — сразу планировать вариант B.

### Вариант D — HSV agreement filter (postprocessing страховка)

**Идея:** в SGIE postprocessing C++ parser добавить sanity-check: после CNN softmax считать долю пикселей предсказанного цвета в crop'е через HSV histogram, и отбрасывать классификацию когда эта доля ниже порога. Концепция повторяет легаси F2 фильтр («CNN + HSV agreement»), который существовал в `api/legacy_pipeline.py` + `pipeline/analyzer.py` до миграции на DeepStream.

| Параметр | Значение |
|---|---|
| **Где** | SGIE postprocessing C++ parser в `libnvdsinfer_racevision` (`color_classifier_parser.cpp`) — после `NvDsInferClassiferParseCustomColor` softmax вычисления |
| **Что** | После CNN softmax: посчитать % пикселей предсказанного цвета в crop через HSV histogram (например `H ∈ [50°, 70°] ∧ S > 0.3` для green), при доле < порога — отбросить классификацию (вернуть `unknown` или `color_id=255`) |
| **Time** | 1-2 вечера (включая калибровку порогов на наших 420 размеченных crops) |
| **Effect** | Убирает 80%+ false positive: зрители на газоне, тёмные люди, любые ракурсы где доминирует фон — потому что фактический % «нужного» цвета в torso-области малый. **КАМАЗ-подобные случаи** (объект реально нужного цвета занимает крупную долю crop'а) **НЕ ловит** — это ограничение метода, для них нужен B или C. |
| **Risk** | Ложно отбрасывает жокеев в крупных bbox с большим фоновым margin (далёкие камеры). Митигация: подбор порогов на размеченном наборе — у нас есть **420 crops с ground truth** (`labels.json`), можно откалибровать на них и получить ROC по precision/recall. |
| **Совместимость** | Не блокирует A/B/C — это независимый слой postprocessing'а. Можно складывать с любым из них (D + A, D + B). |
| **Ничего не ретрейнится** | ✅ Прод-веса не трогаем |

## 6. Что НЕ делать (явно — для будущих сессий)

- ❌ **Не «доучивать»** color_classifier_v4 на новых данных без класса `not_jockey`. Симптом сместится (вместо green будут другие цвета), root cause останется.
- ❌ **Не лечить confidence threshold** в downstream pipeline («если conf < 0.85 → ignore»). На наших данных 22% not_jockey имеют conf ≥ 0.9 — порог нужно ставить >0.95, а тогда мы потеряем real jockeys в сложных ракурсах (где они вполне получают conf 0.85-0.92).
- ❌ **Не переписывать архитектуру в C++** не имея ретрейненной модели в `.pt`. Сначала прототип в Python, потом перенос.

## Update 2026-04-25: Variant D эмпирически опровергнут

- HSV calibration выполнена на 420 размеченных crops
- Best result: rejected_not_jockey=97.6%, tp_rate=84.1% (criteria 70%/90% не достигнут)
- Главная находка: median Hue для всех 4 групп (green/yellow/red/not_jockey) лежит в узком диапазоне [17-22] из-за warm-temperature shift от освещения камеры. Hue не дискриминирует классы вообще.
- Подтверждение: ни один true red jockey не имеет hue в [0,10]∪[170,179] (стандартный red range). Аналогично green.
- Артефакты: `hsv_calibration/grid_search.csv`, `gallery_lost_greens_diag.html`, `hue_distribution_per_label.png`

Вывод: классификация цвета на основе hand-crafted color features на этих данных невозможна. Следующий шаг — Variant E (embedding-based metric learning), требует 3-5 вечеров и сбора расширенного датасета.
